## 4. Staging plan (the gate you asked for)

There is no separate staging environment. Substitute:

1. `pg_dump` prod DB to a local file.
2. `psql` restore to a fresh shadow DB on the same Vultr Postgres
   managed cluster (new schema name or new DB — either works).
3. Set a local `.env.staging` with `DATABASE_URL` → shadow DB.
4. Run `prisma migrate deploy` against shadow.
5. Verify with SQL: `SELECT COUNT(*) FROM memberships;` should equal
   `SELECT COUNT(*) FROM companies;`. Role should be `OWNER` on all.
6. Build the web app locally against the shadow DB.
7. I open an SSH tunnel so you can hit `localhost:3000` from your
   browser and log in as yourself.
8. Verify:
   - Both of your existing organizations load at `/organizations`.
   - All issues visible on dashboard.
   - Issue detail pages load.
   - Telegram channel still connected on the settings page.
9. If OK → apply the same migration to prod + deploy the new code.

Rollback plan if prod breaks:
- `prisma migrate resolve` + restore `companies.owner_id` from the
  `pg_dump` snapshot.

- The migration is additive for the new tables (memberships,
  audit_log) and a column DROP for `companies.owner_id` — the
  dropped column contents live in the snapshot, not elsewhere, so
  the snapshot restore IS the rollback.

## 5. Open questions — ANSWERED per spec-update msg

### Q1 Issue creator semantics → split into three fields
- `detectedBy` enum: `human` / `bot` / `inbound_telegram` /
  `inbound_email` / `system`
- `createdByUserId` nullable FK (null for pure bot detection)
- `assignedToUserId` nullable FK (stays null in Commit A; routing
  logic populates it in v1.6+)

**Migration impact:** rename `Issue.ownerId` (currently @map to
`owner_id`) to `createdByUserId`; add `detectedBy` + `assignedToUserId`
columns. Replaces the simpler rename in the initial migration — update
the pending SQL file before pushing to prod.

### Q2 Cross-org auto-routing → auto-route across any membership role
- Matcher routes to any org the forwarding user is a member of
  (OWNER / ADMIN / MEMBER — any role).
- Issue carries four source fields:
  `sourceChannel`, `sourceUserId`, `matchedByIdentifier`,
  `matchedToOrganizationId` + `matchConfidence`.
- If match lands on an org the user is NOT a member of → **"needs
  manual routing"** queue (nullable `organizationId` on Issue +
  a new column `needsManualRouting: boolean` or status flag).
- Audit entry fires on the **destination** org's trail, not the
  source user's.

**Migration impact:** add the four source columns + the
needs-manual-routing flag. These land in Commit A since they gate
the matcher rewrite.

### Q3 Delete-org → soft delete only, OWNER gated
- Add `deletedAt` + `deletedByUserId` to Organization.
- Only OWNER triggers.
- Related-org rules:
  - Soft-delete does NOT cascade to related orgs.
  - `parent_subsidiary` edges flip `status: 'stale'` on either
    side's soft-delete, not removed.
  - Stub orgs (`source='inferred_from_partner_list'`) — their
    edges are **hard-removed** on soft-delete; stubs are ephemeral.

**Migration impact:** `deletedAt`, `deletedByUserId` columns now.
`OrganizationRelationship` + `Organization.source` land in v1.5
Phase A, not here — but the delete logic references both, so build
the soft-delete action to no-op the edge updates gracefully when
those tables don't exist yet.

### Q4 Membership uniqueness → confirmed
`@@unique([userId, organizationId])`. Role changes UPDATE the existing
row. Already in the schema.

### Q5 Dashboard scope → aggregate NOW, active-org switcher is v1.4.1 blocker for v1.5
- Commit A ships aggregate across all user's orgs (current behavior).
- **v1.4.1** (between Commit A prod + v1.5 Phase A) must ship the
  active-org switcher.
- Design `listUserOrgIds(userId, { activeOrgId? })` so the switcher
  is a **one-line filter**: `activeOrgId ? [activeOrgId] :
  <all user's orgs>`. Implementing this now saves a follow-up
  refactor.
