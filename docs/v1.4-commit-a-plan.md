## 4. Staging plan (the gate you asked for)

There is no separate staging environment. Substitute:

1. `pg_dump` prod DB to a local file.
2. `psql` restore to a fresh shadow DB on the same Vultr Postgres
   managed cluster (new schema name or new DB — either works).
3. Set a local `.env.staging` with `DATABASE_URL` → shadow DB.
4. Run `prisma migrate deploy` against shadow.
5. Verify with SQL: `SELECT COUNT(*) FROM memberships;` should equal
   `SELECT COUNT(*) FROM companies;`. Role should be `OWNER` on all.
6. Build the web app locally against the shadow DB.
7. I open an SSH tunnel so you can hit `localhost:3000` from your
   browser and log in as yourself.
8. Verify:
   - Both of your existing organizations load at `/organizations`.
   - All issues visible on dashboard.
   - Issue detail pages load.
   - Telegram channel still connected on the settings page.
9. If OK → apply the same migration to prod + deploy the new code.

Rollback plan if prod breaks:
- `prisma migrate resolve` + restore `companies.owner_id` from the
  `pg_dump` snapshot.
- Revert the branch merge; redeploy previous main.
- The migration is additive for the new tables (memberships,
  audit_log) and a column DROP for `companies.owner_id` — the
  dropped column contents live in the snapshot, not elsewhere, so
  the snapshot restore IS the rollback.

## 5. Open questions left for you

1. **Issue.createdById semantics.** Currently every issue has a
   creator = the user whose Telegram received the notice. With
   multi-user, should a member who forwards a notice THEY received
   via THEIR Telegram connection create an Issue owned by THEIR
   membership's org? (Yes, I think — but flag for confirmation.) Or
   should there be an "assigned to" concept separate from creator?
2. **Cross-org data leakage on matcher.** Bot receives a notice from
   user X's Telegram. X is a member of org A and org B. Notice
   references CR `1234567890` which matches a company registered in
   org B. Do we auto-route to org B, or require X to be an owner of
   B to auto-route there? (Auto-route for now; audit row records
   the decision so it's reviewable.)
3. **Delete-org cascade.** Spec §7 says `owner: everything,
   including delete org`. Cascade behaviour: delete all issues +
   docs + messages + memberships? Or soft-delete (set `isActive=
   false`)? Current schema has `isActive` already — recommend soft
   delete, preserve audit trail.
4. **Membership uniqueness.** Spec implies one membership per
   (user, org). Enforced via `@@unique([userId, organizationId])` in
   Commit A. Ok?
5. **Dashboard scope.** Currently dashboard shows metrics across
   ALL of a user's companies (the old model). With multi-org, do
   we:
   (a) keep showing aggregate across every org the user is a
       member of, or
   (b) add an "active org" concept (dropdown in header), and every
       page scopes to that org only?
   Most B2B tools go with (b). Big UX decision; spec doesn't say.
   Recommend (b) for the follow-up commit — Commit A can ship with
   (a) unchanged.
