## 1. Inconsistencies between the spec and the current code

### 1.1 DocumentType enum — 28 → 7, with renames

28 values. Spec asks for 7. Overlap/renames:

| `ZATCA_VAT_CERTIFICATE` | `VAT_CERTIFICATE` | Rename. |
| `CHAMBER_OF_COMMERCE` | `CHAMBER_MEMBERSHIP` | Rename. |
| `GOSI_REGISTRATION` | `GOSI_REGISTRATION` | Match. |
| *(new)* | `MOCI_REGISTRATION_PACKAGE` | New. This is the SBC-issued aggregated doc. No current type covers it. |
| *(new)* | `ARTICLES_OF_ASSOCIATION` | New. |
| *(new)* | `NATIONAL_ADDRESS_PROOF` | New. |
| ~~21 other types~~ | *(dropped)* | Balady, SFDA, Mudad, Muqeem, CST, SAMA, CMA, MISA, MOI/MOJ/MOH/MOT/MOE/MOC/MOS licenses, SASO, Monsha'at, SOCPA, SCE, MHRSD Nitaqat, Zakat Assessment, Other — all gone from extraction. **Open question: these were in the onboarding UI ("upload any of these 28 docs"); dropping them from extraction ≠ dropping them from storage. The 21 dropped types matter because `ALL_DOC_TYPES` in
`AddDocumentDialog.tsx` is the user-facing "add another doc" picker and
`DOC_META` in `DocumentUploadGrid.tsx` is the descriptive map. Both
reference the dropped types. Current behavior: users upload, extractor
runs a type-specific prompt, fields land on `Company`. If we drop
those types, users can't upload them anymore — unless we keep them as
"OTHER" pass-through with manual-entry fallback.

### 1.2 Extraction approach — OCR+text-LLM → vision-direct
Current: `OCR (GCP Vision) → OpenAI gpt-5-mini (text prompt)`. Spec:
`rasterize → PNG per page → Claude Sonnet 4+ vision`.

Dependencies needed (not installed):
- `@anthropic-ai/sdk` — not in any package.json.
- `pdf-to-img` is already in `packages/core/package.json` (unused since
  we dropped pdfjs). Usable for rasterization. Alternative:
  `pdf2pic` (calls GraphicsMagick/Ghostscript), requires system
  binaries on the Vultr VPS. `pdf-to-img` is pure JS, no binaries.

Env not set: `ANTHROPIC_API_KEY` is absent from `/opt/makyn/.env`
(confirmed via SSH earlier). **Blocker for Phase B.**

### 1.3 Company identifier columns
Spec: three indexed columns — `unified_number` (10 digits starting 7),
`cr_number` (10 digits), `national_number`. Current `Company` model
has `crNumber` only (`@unique`), no unified_number, no national_number.

Spec also says "inbound notices can reference any" — important: the
notice-matching code in the bot (`packages/core/matching/company-matcher.ts`)
currently keys on `crNumber`, `zatcaTin`, `gosiEmployerNumber`,
`qiwaEstablishmentId`. **Open question: add `unifiedNumber` and
`nationalNumber` to the matcher's keyed-identifier set?** — probably
yes, but the spec doesn't say explicitly.

Validation rules in the spec:
- `unified_number`: 10 digits starting 7.
- `zatca_number`: 15 digits. **Spec says "15 digits" but doesn't say
  "starting with 3" — the current extractor prompt in
  `extractor-prompts.ts:60` says "15 digits starting with 3". Confirm.**
- `gosi_number`: 9 digits. **Current prompt says "typically 9–11
  digits" (`extractor-prompts.ts:12`). Sample docs will show which is
  right.**

### 1.4 ExtractionAudit table — duplicate of spec's v1.4 AuditLog?
Spec v1.3.1 asks for:
```
extraction_audit (
  document_id, document_type, rasterized_image_hash,
  prompt_version, extracted_json, model_version,
  timestamp, user_id
)
```
Spec v1.4 asks for a polymorphic:
```
audit_log (
  id, actor_user_id, organization_id, entity_type, entity_id,
  action, diff JSONB, ip, user_agent, timestamp
)
```
**These should not be the same table.** `extraction_audit` is
per-extraction forensic data (image hash, raw JSON, model version) —
these don't fit cleanly in the generic `diff` field. Recommendation:
keep `extraction_audit` as a separate table dedicated to extraction
forensics. `audit_log` handles member actions. Both coexist.

The current codebase has a *third* AuditLog table (the one we have)
used for AI pipeline logging (`extractor.ts:224`). Rename-or-replace
decision needed — it's the same name the v1.4 spec uses.

### 1.5 Existing Company fields that become extraction-derived
Current `Company` schema has ~50 optional fields (Balady, SFDA, MOH,
tourism, CST, SAMA, etc.) that were meant to be filled via the 28
onboarding doc types. With v1.3.1's narrower extraction scope, these
fields become: (a) filled by manual entry, or (b) filled by uploading
the relevant cert and having it routed to an `OTHER` extraction that
the user reviews. **Spec doesn't clarify which.** If (b), we need a
plan for the 21 dropped doc types beyond "skip".

## 2. Open questions before Phase B can execute

1. **Model choice.** `claude-sonnet-4-6` (faster/cheaper) vs
   `claude-opus-4-7` (better on low-quality Arabic scans). Cost per
   extraction matters at volume. For dogfooding, default Sonnet. But
   need explicit call.
2. **Rasterization DPI.** 150dpi is fine for born-digital PDFs; 200–
   300dpi needed for scans. Higher DPI = bigger images = more input
   tokens = more $$$. Fixed DPI, or tiered (start at 150, retry at 300
   on low confidence)?
3. **Page limit per document.** Articles of Association can run
   10–25 pages. Vision API token cost scales linearly. Cap at
   first-N pages (which N?) or send all?
4. **Type-declaration at upload.** Spec says user declares type.
   Current UI has a "select from 28" dialog. Replace with 7-type
   radio group? If user selects wrong type, spec says "retry with
   different doc type" — that implies the dropdown stays editable
   after upload. Ok.
5. **Auto-detect as safety net.** When type declared is obviously wrong
   (user uploads CR claiming it's AoA), should the prompt probe
   whether the declared type matches and refuse? Spec says "verbatim
   only, return null if not present" — but silently returning all
   nulls is the worst failure mode. Probably need one preamble
   question: "Is this document a {declared_type}? If not, return
   `{"document_type_mismatch": true}`."
6. **Merge — what wins on conflict?** Spec: "later uploads enrich,
   don't overwrite non-null". So first write wins. That means: if the
   first-uploaded CR has the wrong Arabic name (OCR garbage that
   passed extraction), subsequent correct docs won't fix it. Need a
   "force overwrite" path in the conflict-review UI.
7. **PDPL retention for extracted_json.** Extraction audit stores raw
   extracted JSON — which contains full name_ar, national IDs, iqama
   numbers. How long do we retain? 90 days? 1 year? Lifetime of org?
8. **Articles of Association → which amendment.** AoA have
   amendments (`ملحق عقد التأسيس`). Extraction should run on the
   LATEST amendment (partners list is restated). How does the user
   tell us which upload is the latest? Add an "amendment date" field
   at upload time, or extract it from the doc and sort?
9. **Entity type enum.** Spec has `entity_type` / `entity_subtype` on
   MOCI. Common Saudi types:
   - `شركة ذات مسؤولية محدودة` (LLC)
   - `شركة مساهمة` (Joint Stock)
   - `شركة تضامن` (Partnership)
   - `شركة توصية بسيطة` (Simple Limited Partnership)
   - `مؤسسة فردية` (Sole Proprietorship)
   - `شركة شخص واحد` (Single-Person Company)
   - `شركة توصية بالأسهم` (Partnership by Shares)
   Store as enum or free-text? Free-text is forgiving; enum enables
   filtered dashboards. Probably free-text for v1.3.1 with a
   normalizer that maps to an enum later.

## 3. Edge cases in document-generation detection

### 3.1 Commercial Registration — 3 generations
- **Gen 1 (pre-2015)**: MOCI paper certificates, often scanned. No
  QR, no unified_number, sometimes hand-annotated. OCR quality varies
  wildly with scan quality. Only `cr_number` is extractable. National
  number and unified_number **WILL BE NULL** on these — don't treat
  nulls as extraction failure.
- **Gen 2 (2015–2021)**: MOCI online-issued PDF. No QR, no
  unified_number yet (introduced 2021). Standardized layout, good
  OCR. `cr_number` + most of the CR fields extractable.
- **Gen 3 (2021+)**: Saudi Business Center (SBC) issued. Has QR code
  in bottom-right, contains `unified_number` (10 digits starting 7),
  `national_number`, possibly `linked_services` block. **This is the
  generation the MOCI_REGISTRATION_PACKAGE type targets.**

The spec's schema for `CR_CERTIFICATE` has `national_number` — Gen 1
and most Gen 2 certs won't have this. **Prompt must explicitly
tolerate absence; validator must not flag null national_number as
suspicious for Gen 1/2.**

Detection heuristic: if doc has a QR code + phrase `الرقم الموحد`,
it's Gen 3. If no QR, Gen 2 or earlier.

### 3.2 VAT Certificate — GAZT vs ZATCA
- **GAZT-era (pre-June 2021)**: Letterhead says
  `الهيئة العامة للزكاة والدخل`. Code expects "ZATCA" — will
  mis-detect if we key on the letterhead text.
- **ZATCA-era (June 2021+)**: Letterhead says
  `هيئة الزكاة والضريبة والجمارك`.

TIN format unchanged (15 digits starting 3). Risk: extractor prompt in
`extractor-prompts.ts:44` uses "Saudi ZATCA VAT Registration
certificate" — an old GAZT cert might confuse the LLM if we hard-code
"ZATCA" in the prompt. **Prompt should say "Saudi VAT registration
certificate (GAZT or ZATCA)".**

### 3.3 Articles of Association — amendments
Primary AoA is a notarized document listing partners, capital, and
purposes. Every change (partner added, capital increased, activities
amended, address changed) produces a separate appendix document
(`ملحق عقد التأسيس رقم N`). Partners on the original AoA may not be
current. Signatures on the original may not be current either.

Risks:
- User uploads the original AoA only (most common — they don't know
  appendices are separate docs). Extracted partners[] is stale.
- User uploads the latest appendix only. It references the original
  but the extraction will miss partners who weren't changed in this
  amendment (the amendment only lists what CHANGED).
- User uploads multiple amendments; we need to reason about order.

**Recommendation:** ask user at upload "Is this the original AoA or
an amendment?" and "If amendment, what date?" Sort by date; latest
wins for conflicting fields. **Spec doesn't address this — flag.**

### 3.4 National Address — format change ~2022
- **Pre-2022**: Only Arabic fields (`short_code`, `building_no`,
  `street_ar`, `district_ar`, `postal_code`, `city_ar`,
  `secondary_no`).
- **Post-2022**: Adds English fields (`street_en`, `district_en`,
  `city_en`).

Spec includes both ar and en. Gen-1 extraction will produce null
English fields — not an error.

`short_code` format is 4 letters + 4 digits (e.g. `RCKA1234`), not
just digits. **Prompt should specify alphanumeric.**

## 4. Risk areas for silent extraction failures

Prioritized — top of list is most dangerous.

1. **Arabic text rendering as PUA (Private Use Area) code points.**
   Subsetted Arabic fonts in PDFs (common for government docs) use
   PUA glyph IDs that don't map to Unicode letters. Text-extraction
   pipelines return garbage like ``. Our current pipeline is
   OCR-based (GCP Vision), which re-reads pixels — should be immune.
   But if anyone touches a text-extraction fallback, this re-appears.
   **Mitigation for Phase B: assert that extracted `name_ar` is
   composed of Arabic Unicode block characters before accepting.**

2. **LLM hallucinates from context when OCR returns empty/garbage.**
   Even with empty input, the LLM may produce plausible-looking field
   values based on the declared document type. Extraction confidence
   doesn't reliably signal this — the LLM is confident. **Mitigation:
   gate the LLM call on `ocrText.length > threshold` (e.g. 200 chars
   for a CR); reject with `extraction_failure: insufficient_ocr`
   otherwise.** Current code caps to 12000 but doesn't floor.

3. **Hijri → Gregorian date conversion off-by-one.** Hijri calendar
   varies by sighting; canonical conversion tables differ by a day
   between Um Al-Qura and Kuwaiti algorithms. If only Hijri is
   printed, converted Gregorian may be off. **Mitigation: always
   store the raw Hijri string alongside converted Gregorian. On
   conflict with a doc that has both, trust the printed Gregorian.**

4. **unified_number vs cr_number swap.** Both are 10 digits. LLM
   may confuse them on docs that print both with similar labels.
   **Mitigation: validator asserts `unified_number[0] === '7'`;
   reject and ask for manual entry if not.**

5. **Capital in Arabic-Indic digits vs Western digits.**
   `١٠٠,٠٠٠` (Arabic-Indic) may or may not be normalized by OCR
   depending on Vision config. LLM may interpret
   `١٠٠,٠٠٠ ريال` as `100000` SAR or as 100.000 SAR (comma as
   decimal). **Mitigation: post-extraction validator normalizes
   digits, rejects decimal commas.**

6. **Partial extraction of multi-page docs.** If Vision OCR times
   out on page 4/5, we get pages 1–3. Extraction has no signal it's
   incomplete; confidence reports on what it saw. **Mitigation:
   compare page count declared by the PDF vs pages returned by
   Vision; flag mismatch.**

7. **Scanned photo of a screen showing a PDF.** Low res, glare,
   skew. OCR partially succeeds. Extraction returns partial data
   with medium confidence. User accepts. **Mitigation: at upload,
   warn if file is a JPEG >10MP taken by phone (EXIF) and suggest
   re-scanning.**

8. **OCR'd text exceeds 12,000 char cap** (`document-extractors.ts:74`).
   Long AoA gets truncated mid-document. Later sections (often where
   authorized signatories live) are silently dropped. **Mitigation:
   switch to vision API which doesn't have this constraint in Phase
   B; for Phase A transitional code, bump the cap to 30k or
   chunk-and-merge.**

## 5. Assumptions in the spec that sample docs may contradict

(These are hypotheses to test against your actual PDFs.)

1. **MOCI_REGISTRATION_PACKAGE always has `linked_services` block.**
   Companies with no employees won't have GOSI. Companies with no
   physical location won't have Balady. Single-activity companies may
   not have HRSD. **Spec already says null for "لا يوجد" — good, but
   the linked_services object itself may be absent from the PDF if
   there are zero linked services. Prompt must tolerate missing
   block.**

2. **MOCI has an `address` sub-object with structured fields.**
   Older MOCI packages show the address as a free-text line. The
   structured breakdown (building_no, street_ar, secondary_no,
   postal_code, city) only exists when the company also has a
   National Address. For CR-only companies (no NA registered), MOCI
   will have city + maybe street, nothing else. **Prompt must be
   resilient to partial address.**

3. **`managers` array.** Spec assumes >=1 manager. LLCs have
   managers (`مديرون`); sole proprietorships have an "owner"
   (`مالك`), not a manager. Treating them as the same field will
   cause confusion. **Rename to `representatives` or split.**

4. **AoA `partners` have `shares` + `ownership_pct`.** Some old
   AoA only state `ownership_pct` without the share count; some
   state shares only (for ~equal partners with implicit pct). **Both
   nullable, not both required.**

5. **AoA `authorized_signatory` is singular.** LLCs can have
   multiple authorized signatories, each with different powers (joint
   vs several). **Should be an array of objects with a `powers`
   text field.**

6. **NA `short_code` is present on all docs.** Older (pre-2022) NA
   docs may only have `building_no` + `postal_code` + `district` —
   no short_code. **Nullable.**

7. **`cr_expiry_date_hijri` on MOCI Registration Package.** Some CRs
   issued under the "indefinite validity" reform don't have an
   expiry date at all (they auto-renew with annual fee). Field must
   be null, not default to a date in the far future.

8. **Capital always in SAR.** True for Saudi-registered entities.
   But amounts may be printed as `١,٠٠٠,٠٠٠ ريال سعودي`,
   `SAR 1,000,000`, `1 مليون ريال`, or `10 مليون ريال سعودي`.
   Prompt must normalize.

## 6. Recommendations for the Phase B prompt structure

Not asked but useful as you review. Each doctype prompt should have:

```
<role>
You extract structured fields from a single Saudi government
document. You will be shown a rendered image of one or more pages.
</role>

<rules>
- Extract verbatim. Do not translate. Do not infer.
- Return null for any field not visibly present in the image.
- If the document is not a {DECLARED_TYPE}, return
  {"document_type_mismatch": true, "actual_type_guess": "..."} and
  nothing else.
- Arabic text: return exactly as printed, preserving diacritics.
- Dates: return the printed form. If Hijri, include the Hijri
  original AND your Gregorian conversion in a sub-object.
- Numbers: normalize Arabic-Indic digits to Western (0-9).
- IDs: no spacing, no separators — 10 consecutive digits for
  cr_number/unified_number/national_number, 15 for zatca.
</rules>

<output_schema>
{ ...type-specific schema with every field explicitly nullable... }
</output_schema>

<quality_signals>
- "confidence": 0.0-1.0
- "notes": any ambiguity, partial scan, rotated page, illegible
   sections
- "pages_processed": number of distinct pages you saw
</quality_signals>
```

## 7. Ordered punch-list for Phase A (non-blocking, ships without the key)

These are actually doable now without `ANTHROPIC_API_KEY`:

1. `Company` schema: add `unifiedNumber` (unique), `nationalNumber`
   (indexed, not unique — could collide with owner's national ID on
   sole-prop). Migration backfills null.
2. `ExtractionAudit` table (separate from v1.4 AuditLog). Migration
   creates empty.
3. DocumentType enum: add `MOCI_REGISTRATION_PACKAGE`,
   `ARTICLES_OF_ASSOCIATION`, `NATIONAL_ADDRESS_PROOF`. Rename
   others as decided.
4. Zod schemas + TS types for each of the 7 doctype extraction
   results (pure types, no runtime dependency).
5. Merge layer: `mergeExtractionsForOrg(extractions) →
   OrganizationProfile` with enrich-not-overwrite + conflict flag.
   Unit-testable in isolation.
6. Validators: `isValidUnifiedNumber`, `isValidZatcaTin`,
   `isValidGosiNumber`, `isValidCrNumber`, `isValidNationalId`. Pure
   functions.
7. UI: new 7-type radio group at upload. Manual-entry fallback
   always visible. "Retry with different type" button on declared-
   type mismatch.
8. Route the current OpenAI-based extraction through the new
   merge/validator layer so Phase B's vision call is a drop-in
   replacement.

All of the above is pure schema + TS. Phase B (vision wiring) lands
on top once key + model are chosen.
